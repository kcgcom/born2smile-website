# 콘텐츠 기회 평가 모델

현재 활성 모델은 `opportunity-v1.9`이다. 검색 추이와 연관 키워드 검색량은 점수에 사용하지 않으며 참고 근거로만 제공한다.

## 원본 신호

- `demandScore`: SearchAd 월 검색량의 전체 택소노미 백분위 70%와 카테고리 내부 백분위 30%. 단, 서브그룹이 4개 미만인 카테고리는 작은 표본의 양극화를 막기 위해 전체 백분위만 사용한다. 검색량이 없으면 `null`이다.
- `contentGapScore`: 대표 콘텐츠의 주제 직접성·본문 깊이·택소노미에서 자동 추출한 개념군 커버리지·최신성으로 계산한 콘텐츠 공백이다. 정확한 주제 구문과 같은 문맥에 흩어진 토큰 일치를 구분하며, 제목·소제목·정확 구문 없이 요약·본문에서만 간접 매칭된 글은 보조 근거 상한을 적용한다. 후속 글은 새로운 개념을 보완할 때만 가산하며 검색량은 포함하지 않는다.
- 콘텐츠 공백은 `55 이상` 큰 공백, `25~54` 부분 공백, `25 미만` 대체로 충족으로 해석한다. 미충족 검색 수요에는 실제 신규 글 평가 대상인 큰 공백만 합산한다.
- 브랜드검색은 블로그 콘텐츠 공백 평가 대상에서 제외하고 전체 검색 수요에만 유지한다. 근거에는 대표 게시글의 매칭 위치·본문 블록 수·인식 개념·후속 글의 신규 개념 기여를 표시한다.
- `patientBusinessValue`: 환자 유용성 신호와 상담 근접성 신호를 같은 비중으로 결합한다.
- `strategicFit`: 병원의 진료 전략과 카테고리의 적합성이다.
- `pageGapScore`, `faqGapScore`: 실제 진료 페이지와 FAQ가 검색 주제에 필요한 내용을 포함하는지 평가한다.

페이지·FAQ는 해당 주제가 실제 진료 페이지 범위에 속할 때만 평가한다. 페이지 본문 커버리지가 40 미만이면 FAQ를 먼저 추가하지 않고 페이지 보강을 우선한다. 페이지 본문 커버리지가 40 이상이고 FAQ 커버리지가 70 미만이며 질문 신호가 60 이상일 때만 FAQ 후보가 된다. 진료 범위 예외는 `TOPIC_ACTION_OVERRIDES`에서 보수적으로 관리한다.

분석 전용 `기타 진료`의 `턱관절/이갈이`, `발치/사랑니`, `잇몸재생`은 보조 진료 환자 교육 주제다. 새 글과 전체 FAQ 커버리지를 기준으로 한 FAQ 보강만 허용하며, 전용 진료 페이지 보강에는 포함하지 않는다. 공개 콘텐츠는 턱관절·사랑니는 `건강상식`, 잇몸재생은 `예방관리`에 발행한다.

건강상식의 `증상/내원판단`은 여러 진료 영역에 걸친 환자 안내 주제다. 블로그와 전체 FAQ 보강만 허용하며 특정 진료 페이지 보강에는 포함하지 않는다.

치과선택의 `브랜드검색`은 홈페이지·병원소개·연락처가 담당하는 탐색 의도이므로 신규 블로그·페이지·FAQ 추천을 생성하지 않는다.

페이지 평가는 주제별 근거를 유지하되 실행 계획은 대상 페이지별로 통합한다. 각 주제는 `covered`, `promote-from-faq`, `missing`, `needs-confirmation`, `not-applicable` 중 하나로 분류한다. `missing`과 `promote-from-faq`만 페이지 계획 점수에 포함하며, `needs-confirmation`은 점수에 넣지 않고 작업 전 확인 항목으로 표시한다. 코드의 실제 페이지 내용만으로 판별하기 어려운 범위는 `PAGE_STATUS_OVERRIDES`에서 명시적으로 관리한다.

## 행동별 가치

```text
신규 글 = 검색 수요 35 + 콘텐츠 공백 35 + 환자·사업 가치 20 + 전략 적합성 10
페이지 = 페이지 공백 35 + 환자·사업 가치 25 + 검색 수요 25 + 전략 적합성 15
FAQ = 질문 신호 25 + FAQ 공백 25 + 검색 수요 15 + 환자·사업 가치 20 + 전략 적합성 15
페이지 통합 계획 = 가장 높은 페이지 주제 가치 70 + 차상위 최대 2개 평균 30
```

페이지 계획의 식별자는 주제명이 아니라 `page:{targetPage}`를 사용한다. 따라서 같은 진료 페이지에 여러 공백 주제가 있어도 플래너에는 한 작업만 만들어지고, 브리프와 근거 스냅샷에는 기여 주제와 확인 필요 주제가 함께 저장된다.

검색량이 없으면 신뢰도 `C`, 행동 자격 `needs-data`로 분리한다. 규칙 기반 전체 본문 분석을 완료한 정상 평가는 신뢰도 `B`다. 향후 의미 분석 공급자를 추가하더라도 행동별 가치의 데이터 계약과 버전 규칙은 유지한다.

## 실행 순서

기본적으로 행동별 가치가 높은 후보를 먼저 제시한다. 가치 차이가 10점 이내면 작업 비용이 낮은 후보를 우선하며, 같은 유형이 이미 3개 선택된 경우 10점 이내의 다른 유형 후보로 편중을 완화한다. 관리자가 기회분석에서 직접 선택한 후보는 첫 번째로 제시한다.

## 검증

- `pnpm verify-opportunity-scoring`: 고정 사례와 점수 계약 검증
- `pnpm report-opportunity-scoring`: 활성 택소노미·검색량·게시글 전체 본문을 이용한 실제 후보 분포 출력

가중치나 자격 조건을 변경할 때는 `OPPORTUNITY_MODEL_VERSION`을 올리고 두 검증 명령의 결과를 검토해야 한다.

## 택소노미 배포

코드의 `CATEGORY_KEYWORDS` 변경은 DB 활성 택소노미를 즉시 덮어쓰지 않는다. 관리자 검색 트렌드의 후보 검토 화면에서 `코드 변경 적용 준비`를 실행하면 코드 정의가 대기 버전으로 저장된다. 다음 자동 수집 또는 수동 데이터 갱신은 그 버전으로 SearchAd와 DataLab을 함께 수집하며, 두 스냅샷 게시가 성공한 뒤에만 대기 버전을 활성화한다. 수집 실패 시 기존 활성 버전과 스냅샷을 유지한다.

분석 전용 주제는 `getBlogCategoryForKeywordTopic()`으로 공개 블로그 카테고리에 매핑한다. 같은 분석 카테고리라도 서브그룹에 따라 발행 위치가 다를 수 있으므로 분석 카테고리 슬러그를 공개 블로그 URL이나 포스트 저장값으로 직접 사용하면 안 된다.
